# What is MLOPS?

Machine Learning 프로젝트를 하다보면 데이터의 버전관리 문제, 모델의 버전관리, 사용한 패키지와의 dependecy 문제 등 다양한 문제가 발생한다.
또 여러번의 실험 끝에 최적의 결과를 얻었다고 해도, 실제 데이터에서는 잘 맞지 않아 이전에 실험했던 모델이 필요한 경우도 왕왕있지만 로깅을 제대로 해놓지 않아 타협을 해야하는 경우가 생긴다.
나 또한 실제 모델을 실험하거나 프로젝트를 할 때에 모델 자체를 이해하고 당장 성능을 내는데 급급해 이러한 문제를 반복적으로 겪어왔다.
그런데 이러한 문제를 해결할 수 있도록 프로젝트를 효율적으로 관리하기 위한 방법론/철학이 있는데, 그것이 바로 바로 MLops이다.
MLops에 입각한 프로젝트를 진행하기 위해서는 여러가지 tool을 이해하고 익숙해질 필요가 있다.
본 글에서는 어떤 tool을 다루게 될지 전체적인 흐름만 제시해보도록 하고 tool별로 하나씩 시리즈로 글을 쓰고자 한다.

> [패스트 캠퍼스 강의](https://fastcampus.co.kr/data_online_mlops)를 참고하였다.

## 1. 도커와 쿠버네티스

containerization = 컨테이너화 하는 기술, 나의 머신러닝 코드가 pip 버전, os 종류, python 환경에 독립적이도록 모든 정보를 담아놓은 실행환경을 하나의 패키지로 만드는 과정
container = 격리된 공간에서 프로세스를 실행시키는 기술

도커는 containerization을 가능하게 하는 tool이다.
docker를 통해서 다양한 container를 만들어 이를 실행시킬 수 있다.
그런데 다양한 container를 다루기 위해서는 지휘자가 필요하다. 

container orchestration  = 여러개의 container를 지휘하는 것
(ex)메모리를 많이 잡아먹는 container는 memory가 많이 남는 server에 배치, gpu를 많이 잡아먹는 container는 gpu가 많은 server에 배치시킴 등등

docker container를 다루는 여러 container orchestration tool 중 하나가 쿠버네티스이다.

## 2. Data Management: DVC(Data Version Control)

git은 소스코드의 버전을 관리하는 툴이다. 
프로젝트를 하다보면 깃허브에 데이터까지 업로드하기도 하는데 github는 소스코드의 버전관리를 염두에 두고 만든 공간이다보니 100MB의 한계가 있어 대용량의 데이터를 다루기 어렵다. 
그래서 데이터는 구글드라이브나 유료인 gitLFS 등의 스토리지에 저장한다. 
따라서 스토리지에서 데이터를 따로 보관하고 깃허브에 소스코드를 따로 보관하게 된다.
데이터가 있는 스토리지와 소스코드가 있는 깃허브(또는 gitlab, bitbucket)를 이어주면서 데이터의 버전관리도 해주는 것이 DVC이다.
간단한 원리는 원하는 데이터의 스토리지상 주소가 담긴 dvc파일에 git에 올려서 필요할 때 dvc파일에 쓰인 스토리지 상 주소의 데이터를 가져올 수 있도록 하는 것이다.
DVC는 사용법이 git과 거의 비슷한 데이터 버전관리 툴로, 대부분의 스토리지(amazon s3, google drive etc)와 호환되며 
github외에도 깃을 호스팅하는 다른 서버들(gitlab, bitbucket)과도 연동된다.


## 3. Model Management: MLflow

머신러닝 프로젝트를 하다보면 로그를 남겨야 하는 요소들이 굉장히 많다.
소스코드, 평가 metric 결과, 사용한 parameter, model.pkl파일, data ver, 전처리 코드, 전처리된 데이터 등 다양하다.
그런데 이러한 모든 실험을 혼자 하는게 아닌데 저장 스타일이 다 다르다면 사후적으로 스타일을 통일시키느라 해야할 일이 더 늘어나게 된다.
따라서 이러한 로깅을 통일된 방식으로 깔끔하게 해주는 다양한 툴이 생겨났는데, 그 중 하나가 MLflow이다.

## 4. Model serving : Flask, Seldon core

머신러닝 모델을 잘 만들고 나서 서비스에 사용할 수 있도록 배포하고자 한다면 어떻게 해야할까?
모델 serving이란 **모델의 predict 함수를 call하는 logic이 들어간 소프트웨어 개발**이다.
그런데 모델을 개발하는 데이터사이언티스트가 소프트웨어 개발이 익숙치 않은 경우가 많아 모델 구축과 모델 서빙간의 괴리가 나타난다.
특히 이 모델개발과 서빙 업무를 각각 다른 사람이 하는 경우 머신러닝 개발자와 소프트웨어 개발자간의 의사소통이 어려워, docker image가 미친듯이 커지고 커진 이미지 때문에 traffic을 감당하지 못하거나, 배포 후 모델 평가시 소프트웨어 개발자가 모델 평가 방식을 만들거나 모니터링을 할 때 모델에 대한 이해가 부족해 어려움을 겪을수 있다.
특히 이 모델개발과 서빙 업무를 각각 다른 사람이 하는 경우 머신러닝 개발자와 소프트웨어 개발자간의 의사소통이 어려워, docker image가 미친듯이 커지고 커진 이미지 때문에 traffic을 감당하지 못하거나, 배포 후 모델 평가시 소프트웨어 개발자가 모델 평가 방식을 만들거나 모니터링을 할 때 모델에 대한 이해가 부족해 어려움을 겪는다. 따라서 모델 서빙을 손쉽게 할 수 있도록 해주는 seldon core, TFserving 등의 도구들이 최근 등장하고 있다. 
Flask는 파이썬 웹서비스 개발 프레임워크로, 매우 가볍고 확장성과 유연성이 뛰어나 간단한 기능구현에 적합해 대부분의 ML model의 첫 배포에 자주 이용된다.
